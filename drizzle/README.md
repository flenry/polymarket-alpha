# drizzle/meta — Operational Notes

## Migration chain

| idx | tag | description |
|-----|-----|-------------|
| 0 | `0000_misty_thaddeus_ross` | All 8 tables (auto-generated by drizzle-kit) |
| 1 | `0002_partition_trades` | Converts `trades` and `order_book_snapshots` to PARTITION BY RANGE |

## Why there is no `0001_snapshot.json`

`0002_partition_trades` was registered manually in `_journal.json` because drizzle-kit cannot express `PARTITION BY RANGE` declaratively. As a result, there is no corresponding `0001_snapshot.json` for idx 1 — drizzle-kit's snapshot chain ends at idx 0 (`0000_snapshot.json`).

This is intentional. The absence of `0001_snapshot.json` does **not** prevent `drizzle-kit migrate` from applying the file; the migrator checks `_journal.json` and the `__drizzle_migrations` lock table in the DB to determine which files to run.

## Critical operational rule

Never run `drizzle-kit push` or `drizzle-kit generate` against a DB that has already had `0002` applied. After `0002`, `trades` and `order_book_snapshots` are partitioned tables. Drizzle-kit does not understand partitioned schemas and will generate a spurious diff (attempting to recreate the tables without partition logic). Always use raw SQL for any schema changes to these tables after `0002`.

## How to apply migrations

```bash
# Apply all migrations tracked in _journal.json (0000 + 0002):
pnpm db:migrate

# If drizzle-kit cannot apply the partition DDL cleanly, run it directly:
pnpm db:migrate:partitions
```
